  1. VUE D'ENSEMBLE DU PROJET

1.1 Contexte et objectifs

D2S - Digital Sport Solutions est une plateforme SaaS de gestion intégrée pour complexes sportifs et aquatiques, conçue pour répondre aux défis opérationnels suivants :

Problématiques identifiées :

  • Planification manuelle chronophage (jusqu'à 2 jours par cycle)
  • Gestion papier inefficace (documents perdus, mouillés, dispersés)
  • Manque de visibilité en temps réel sur les opérations
  • Communication fragmentée entre équipes
  • Suivi difficile des certifications et conformité réglementaire
  • Absence d'analyse de fréquentation et d'optimisation des ressources

Objectifs de la solution :

  • Réduction de 80% du temps de planification
  • Digitalisation complète des processus opérationnels
  • Visibilité 24/7 sur l'ensemble des installations
  • Traçabilité et conformité réglementaire automatisées
  • Optimisation des ressources humaines et matérielles basée sur les données

1.2 Public cible

Clients primaires :

  • Centres aquatiques municipaux et privés
  • Complexes sportifs multi-activités
  • Salles de fitness et wellness
  • Parcs aquatiques saisonniers

Utilisateurs finaux :

  • Directeurs d'établissements
  • Managers de structures
  • Agents opérationnels (maîtres-nageurs, coaches, etc.)
  • Personnel administratif

1.3 Périmètre fonctionnel

Modules principaux :

  1. Planification du personnel (Core)
  2. Gestion des agents et compétences
  3. Gestion de la structure et des espaces
  4. Gestion du matériel
  5. Suivi de fréquentation
  6. Registres chimiques (spécifique aquatique)
  7. Formulaires personnalisés et rapports
  8. Messagerie et communication
  9. Stockage documentaire (D2S Storage)
  10. Administration et audit

3.1 MODULE PLANIFICATION (CORE)

Priorité : P0 (Critique)

3.1.1 Types de vues de planification

  1. Planification Globale
  • Affichage : Vue calendrier agrégée de toutes les structures
  • Filtres disponibles :
    • Date (jour, semaine, mois)
    • Structure(s)
    • Espace(s)
    • Zone(s)
    • Agent(s)
    • Type d'activité
  • Formats d'affichage : Vue Gantt ou Calendrier
  • Permissions : Admin (RW), Manager (RW), Agent (RO)
  1. Planification par Structure
  • Affichage : Vue détaillée d'une structure spécifique
  • Informations visibles :
    • Activités planifiées avec horaires
    • Agents assignés par activité
    • Zones occupées et disponibles
    • Matériel utilisé et disponible
  • Permissions : Admin (RW), Manager de la structure (RW), Agents de la structure (RO)
  1. Planification Zone / Agent
  • Vue Zone : Occupation par zone avec timeline heure par heure
  • Vue Agent : Planning individuel de l'agent avec toutes ses affectations
  • Permissions : Selon rôle et rattachement

3.1.2 Création et paramétrage d'activités

Informations obligatoires :

  • Nom de l'activité
  • Zone assignée
  • Date et horaires (début, fin)
  • Au moins 1 agent assigné

Informations optionnelles :

  • Description détaillée
  • Récurrence (quotidienne, hebdomadaire, mensuelle, personnalisée)
  • Matériel requis (avec quantités)
  • Capacité maximale (nombre de participants)
  • Notes particulières

Règles de validation :

  • Vérification disponibilité agents (pas de double affectation)
  • Vérification disponibilité matériel (quantités suffisantes)
  • Alerte si agent dépasse heures hebdomadaires planifiées
  • Alerte si agent mineur et respect législation du travail

3.1.3 Planification automatique (Auto-Schedule)

Fonctionnement :

  1. Analyse des créneaux non attribués
  2. Consultation des disponibilités agents
  3. Prise en compte des demandes de congés validées
  4. Respect des heures souhaitées et maximales par agent
  5. Algorithme de répartition équitable

Algorithme de distribution :

POUR chaque créneau non attribué :

  1. Filtrer agents disponibles (disponibilité + pas en congé)
  2. Exclure agents ayant atteint heures maximales
  3. Prioriser agents n'ayant pas atteint heures souhaitées
  4. Affecter en respectant équilibre de charge
  5. Vérifier compétences requises si définies

FIN POUR

Stratégie d'utilisation :

  • L'administrateur définit d'abord les affectations prioritaires manuellement
  • L'Auto-Schedule complète ensuite les créneaux restants
  • Possibilité d'ajustement manuel post-génération

3.1.4 Modèles d'horaires (Templates)

Création de modèle :

  • Enregistrement d'une semaine type complète
  • Choix : avec agents affectés OU structure vide (zones et horaires uniquement)
  • Nommage et catégorisation du modèle

Utilisation de modèle :

  • Sélection du modèle depuis la bibliothèque
  • Choix de la/les semaine(s) de destination
  • Import en 1 clic de tous les créneaux
  • Ajustements possibles avant publication
  • Possibilité de sauvegarder la version ajustée comme nouveau modèle

Bénéfice mesuré : Gain de 80% du temps de planification

3.1.5 Duplication hebdomadaire

Processus :

  1. Sélectionner semaine source (S-source)
  2. Sélectionner une ou plusieurs semaines destinations (S-dest)
  3. Choisir options de duplication :
    • Avec agents : Copie intégrale (zones + agents + matériel + horaires)
    • Sans agents : Copie structure uniquement (zones + horaires + matériel)
  4. Validation et duplication automatique
  5. Ajustements post-duplication possibles

Règles de gestion :

  • Conservation intégrale des paramètres d'activité
  • Vérification automatique des conflits d'agents si duplication avec agents
  • Alerte si matériel insuffisant sur période destination

3.1.6 Disponibilités des agents

Saisie par l'agent :

  • Interface : Grille horaire interactive (jours x créneaux)
  • Granularité : Par demi-journée ou heure
  • Types : Disponible / Non disponible / Préférence
  • Récurrence : Possibilité de définir disponibilités récurrentes (planning type)

Gestion par le manager :

  • Consultation en temps réel des disponibilités
  • Verrouillage temporaire (pendant création planning) pour éviter modifications
  • Intégration automatique dans l'outil de planification

Affichage dans le planning :

  • Code couleur : Disponible (vert), Non disponible (rouge), En congé (orange)
  • Filtrage rapide : "Montrer uniquement agents disponibles"

3.1.7 Demandes de congés et absences

Workflow de demande :

Agent → Soumet demande (date début, fin, type, motif)

↓

Manager → Reçoit notification (email + SMS)

↓

Manager → Approuve ou refuse (avec commentaire si refus)

↓

Agent → Reçoit notification de décision (email + SMS)

↓

SI approuvée → Intégration automatique dans planning (agent indisponible)

Types de congés/absences :

  • Congés payés
  • Maladie
  • Formation professionnelle
  • Absence exceptionnelle
  • Récupération
  • Autre (à préciser)

Statuts :

  • En attente
  • Approuvé
  • Refusé
  • Annulé

Règles de gestion :

  • Solde de congés payés décrémenté automatiquement si approuvé
  • Impossible de créer activité sur période de congé validée
  • Alerte si agent en congé assigné à une activité existante

3.1.8 Postes disponibles (Open Shifts)

Publication d'un poste vacant :

  • Manager crée un créneau "ouvert" (non attribué mais publiable)
  • Définition : Zone, Date, Horaires, Compétences requises (optionnel)
  • Publication → Notification push à tous les agents éligibles

Inscription par l'agent :

  • Agent consulte postes disponibles dans son planning
  • Inscription en 1 clic
  • Notification immédiate au manager

Validation par le manager :

  • Réception notification (email + SMS)
  • Approbation ou refus
  • SI approuvé → Affectation automatique + notification agent
  • SI refusé → Poste reste disponible pour autres agents

Règles de gestion :

  • Vérification automatique : Agent ne dépasse pas heures hebdomadaires
  • Vérification compétences si définies
  • Premier arrivé, premier servi (ou choix manager si plusieurs candidats)

3.1.9 Demandes de remplacement (Shift Swap)

Workflow complet :

Agent A → Demande remplaçant pour son shift

↓

Système → Identifie agents éligibles (disponibles + compétences)

↓

Agents éligibles → Reçoivent notification

↓

Agent B → Accepte le remplacement

↓

Manager → Reçoit notification de demande de validation

↓

Manager → Approuve ou refuse

↓

SI approuvé :

- Planning mis à jour automatiquement (Agent B remplace Agent A)

- Notifications Agent A + Agent B

SI refusé :

- Demande reste ouverte pour autres agents

- Notification Agent B

Critères d'éligibilité automatique :

  • Agent disponible sur le créneau
  • Compétences correspondantes (si définies)
  • Pas de dépassement heures hebdomadaires

Avantages :

  • Réduction absentéisme
  • Autonomie des agents
  • Traçabilité complète

3.1.10 Gestion multi-structures

Principe :

  • Utilisation de "Structures" comme niveaux de séparation
  • Attribution agents à une ou plusieurs structures
  • Permissions granulaires par structure

Configuration :

  • Création de structures distinctes (ex: Centre aquatique, Fitness, Terrains)
  • Affectation managers par structure
  • Affectation agents à structure(s) pertinente(s)

Visibilité :

  • Admin : Voit toutes les structures
  • Manager : Voit uniquement sa/ses structure(s) assignée(s)
  • Agent : Voit uniquement sa/ses structure(s) assignée(s)

Partage de personnel :

  • Agent peut être affecté à plusieurs structures
  • Planning global agent = agrégation de toutes ses structures
  • Gestion des conflits inter-structures automatique

3.1.11 Gestion budgétaire et reporting

Calcul des coûts de main-d'œuvre :

  • Saisie taux horaire par agent (ou taux par type de poste)
  • Calcul automatique : Heures planifiées × Taux horaire
  • Vue prévisionnelle : Coût semaine/mois avant publication

Rapports disponibles :

  1. Rapport hebdomadaire/mensuel :
    • Total heures par agent
    • Total heures par structure/zone
    • Coût total de main-d'œuvre
    • Répartition par type d'agent
  2. Analyse comparative :
    • Évolution coûts période N vs N-1
    • Identification dépassements budgétaires
  3. Export :
    • Format Excel/CSV pour import logiciel paie
    • Format PDF pour reporting direction

Alertes budgétaires :

  • Alerte si planification dépasse budget prévisionnel
  • Alerte heures supplémentaires imminentes (agent proche seuil)

Protocoles supportés :

  • iCal/ICS (standard)
  • Google Calendar API
  • Microsoft Outlook/Office 365 API
  • Apple iCloud Calendar

Synchronisation :

  • Unidirectionnelle : D2S → Calendrier tiers (lecture seule pour agent)

Fonctionnalités :

  • Rappels natifs calendrier (notifications push sur mobile)

  • Accès itinéraire GPS depuis événement calendrier

3.1.13 Notes et informations sur planning

Fonctionnalité :

  • Ajout de notes textuelles sur n'importe quel jour du planning
  • Visibilité : Tous les agents concernés (structure ou activité)
  • Cas d'usage : Événements spéciaux, consignes particulières, alertes

Affichage :

  • Icône distinctive sur le calendrier
  • Consultation détails en 1 clic
  • Notifications push si note ajoutée après publication

Configuration :

Utilisation agent :

3.6.4 Calcul automatique indice de saturation

Formule utilisée :

Indice de Langelier (LSI) = pH - pHs

où pHs (pH de saturation) dépend de : Température, TAC, TH, TDS

Interprétation automatique :

  • LSI < -0.3 : Eau agressive (corrosive)
  • LSI entre -0.3 et +0.3 : Eau équilibrée ‚úì
  • LSI > +0.3 : Eau entartrante

Recommandations automatiques :

  • Affichage actions correctrices
  • Calcul quantités produits à ajouter (basé sur volume bassin)

3.6.5 Dosages correctifs automatisés

Principe :

  • Saisie valeur mesurée hors plage
  • Système calcule écart vs valeur cible
  • Calcul dose produit nécessaire (basé sur volume bassin)

Exemples calculs :

Correction pH :

SI pH trop élevé (>7.8) :

Dose pH- (kg) = (pH mesuré - pH cible) × Volume bassin (m³) × 0.01

SI pH trop bas (<7.2) :

Dose pH+ (kg) = (pH cible - pH mesuré) × Volume bassin (m³) × 0.015

Correction Chlore :

SI Chlore libre < 0.4 mg/L :

Dose Chlore (g) = (0.8 - Chlore mesuré) × Volume bassin (m³) × 1.5

Affichage recommandation :

  • Produit à utiliser
  • Quantité précise (kg, L, ou pastilles)
  • Mode d'emploi (dilution, zone d'ajout)
  • Délai avant re-mesure

  1. SÉCURITÉ ET CONFORMITÉ

9.1 Conformité RGPD

Principes appliqués :

  1. Minimisation des données :
  • Collecte uniquement données nécessaires au service
  • Champs optionnels clairement identifiés
  • Révision régulière données stockées
  1. Droit d'accès :
  • Agents peuvent consulter toutes leurs données personnelles
  • Export JSON/PDF sur demande (interface "Mes données")
  1. Droit de rectification :
  • Agents peuvent modifier leurs coordonnées, photo, etc.
  • Demande modification données RH via formulaire (validation manager/admin)
  1. Droit à l'effacement :
  • Suppression compte sur demande (soft delete initial)
  • Hard delete après 30 jours (ou conservation légale si obligations)
  • Anonymisation données après départ (conservation stats agrégées)
  1. Portabilité :
  • Export complet données personnelles (format machine-readable JSON)
  • Historique planning, compétences, statistiques
  1. Limitation conservation :
  • Données actives : Durée contrat + 5 ans (obligations légales)
  • Audit logs : 5 ans (conformité)
  • Documents RH : Durées légales spécifiques
  1. Sécurité :
  • Chiffrement transport (HTTPS/TLS 1.3)
  • Chiffrement au repos (données sensibles : numéro SS, documents)
  • Accès basés sur rôles (least privilege)
  • Authentification forte (2FA admins)
  1. Traçabilité :
  • Audit logs complets (qui accède à quoi, quand)
  • Notifications accès données sensibles
  • Rapports RGPD disponibles (DPO)

Registre des traitements :

  • Tenu à jour (obligatoire RGPD)
  • Accessible responsable conformité
  • Décrit : Finalités, bases légales, durées conservation, destinataires

9.2 Hébergement et infrastructure

Hébergement en France/UE (recommandé) :

  • Conformité RGPD (données UE restent en UE)
  • Providers suggérés : OVH, Scaleway, Hetzner

Certifications infrastructure :

  • ISO 27001 (Sécurité information)
  • HDS (Hébergement Données Santé) - si données médicales
  • TIER III/IV (Data centers)

Backups géoredondants :

  • Réplication multi-zones (disponibilité)
  • Conservation hors site (disaster recovery)

9.3 Plan de Reprise d'Activité (PRA)

Objectifs :

  • RTO (Recovery Time Objective) : <4 heures
  • RPO (Recovery Point Objective) : <1 heure (pertes max)

Procédures :

  1. Détection incident (monitoring automatique)
  2. Notification équipe technique (alertes)
  3. Évaluation criticité
  4. Activation PRA si nécessaire :
    • Basculement serveurs secours
    • Restauration derniers backups
    • Vérification intégrité données
    • Tests fonctionnels
  5. Communication clients (status page)
  6. Retour normal + post-mortem
